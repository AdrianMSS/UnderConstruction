Bases Datos
No Esquematizadas

Imagine que desea iniciar un nuevo prototipo

¿Cuáles son las primeras preguntas que se hace todo  "maker" al comenzar? ¿Los datos que voy a almacenar son números de tipo entero o flotante? ¿Para guardar la url de una imagen me alcanzará con una cadena de caracteres de longitud 50 o de 20? ¿Tengo la cantidad exacta de variables que deseo almacenar?

Estas son tan solo unos ejemplos de cientos de preguntas similares  relacionadas a la información que se debe guardar en la base de datos, sin embargo estas no surgen al inicio del desarrollo de un nuevo prototipo, debido a que no es lo primordial para probar la funcionalidad del producto.

¿Qué problemas trae esta situación?

Se asigna tiempos considerables en la modificación de la base de datos, al tener que almacenar más, menos y/o diferentes datos,  "crashear" la aplicación que está recibiendo los datos por no haber realizado pruebas previas e incluso se incurre a gastos en expertos de base de datos.

¿Qué tanta importancia se debe brindar?

Depende del tiempo que se tenga a disposición, sin embargo va a beneficiar directamente la agilidad del desarrollo deseado.

¿Cómo se puede solventar este problema?

Se recomienda la utilización de un motor de base de datos que permita estructuras No Esquematizadas.

¿Bases de Datos No Esquematizadas?

Las bases de datos "normales" también llamadas relacionales, tienen la característica que se debe definir una estructura previa, en donde se establecen campos, tipos de datos, relaciones entre datos, entre otros aspectos; lo anterior las hace poco flexibles. Sin embargo existen bases de datos no relacionales, como: FireBase, MongoDB, Cassandra, entre otras; las cuales proporcionan gran flexibilidad para la reducción de trabajo y tiempo de desarrollo; inclusive algunas bases relacionales, como: PosgreSQL y MySQL, han logrado visualizar esta tendencia, dada por el desarrollo de aplicaciones de IoT, Big Data y productos de makers, que pueden no tener conocimientos avanzados en bases de datos, permitiendoles de esta forma utilizar objetos y flexibilizar esta estructura fuera de su naturaleza propia.



¿Qué problema presentan?

A la hora de realizar consultas complejas y actualizaciones de datos, debido a que no poseen relaciones estrictas, puede requerir más trabajo para obtener los resultados deseados, hechos particulares que hacen del mantenimiento, un proceso más tedioso, al no poseer una estructura fija la inconsistencia de datos puede ser una constante.

Sin embargo para prototipado rápido estas no son las principales consideraciones, ya que en entornos de pruebas la información puede ser volátil.

Ejemplo

Suponga que tiene que administrar una base de datos para 19 proyectos diferentes, en los cuales cada uno a primera instancia se deben conectar a un servidor, una vez establecida dicha conexión se procede a enviar diferentes valores, como: humedad, parámetros de geolocalización, estados de variables, url's de imagenes, estados de proximidad, estados de tipo "on/off",  consumos de corriente eléctrica, entre otros. Dado el contexto anterior, no se puede disponer de una estructura de bases de datos rígida, en este contexto una base de datos no esquematizada, parece ser la opción óptima.

Conclusión 

¿Las bases de datos no esquematizadas son la mejor opción para cualquier desarrollo? 

En realidad siempre se va a depender del tipo de aplicación y manejo de datos que se requiera.

¿Estas bases de datos facilitan el prototipado rápido?

Definitivamente, sí.

¿Se tendrán problemas para cambiar de base de datos, posteriormente?

Todo lo contrario, una vez que se haya probado su desarrollo, resulta más fácil definir los datos verdaderamente necesarios y definir la estructura óptima, pudiendo seleccionar si lo mejor es utilizar un tipo de base relacional o no.
